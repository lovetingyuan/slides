历史过程

  • June 2015, ES module的规范正式定义
  • June 2015 – September 2017: 主流浏览器实现了ES模块的实验性支持
  • September 2017：Nodejs 8.5实现了ES模块的实验性支持
  • September 2017 – May 2018：主流浏览器正式支持ES模块
  • October 2018：node团队提出ES模块重构后的支持方案
  • 2019.11.21：node发布13.2.0，正式支持ES模块（仍有提示，因为一些feature仍在讨论和开发中）
  • ......

Node团队提出新方案的三个原则

  1. 完全遵循ES规范 👍
  2. 尽最大可能保持与浏览器的实现一致 👍
  3. 保证现有的commonjs模块完全向后兼容且无需改动 👍

同时支持commonjs 和 ES module 存在什么问题？

  • node如何判断文件的模块类型？
  • cjs和esm如何和谐共存？
  • 模块是同步加载还是异步加载？

node如何辨别模块格式

  • 通过不同的文件后缀，es的模块使用 .mjs 作为后缀，commonjs使用.cjs或者默认.js
  • 对于第三方模块会根据package.json中的type是module还是commonjs来决定（后缀优先）
  • 内部实现上两种模块会使用不同的加载器
  • 还有一些被废弃的idea，比如在代码顶部使用"use module"来表示esm，或者根据有无import，export关键字来判断等

node模块解析和加载的过程

commonjs的加载

  1. require被调用时进行路径分析
  2. 查找模块的目标文件，对于文本代码直接读取文件内容
  3. 解析模块，json的格式会直接json.parse，js会被包装成函数
    (function (exports, require, module,  __filename, __dirname) {
  // code
});
  4. 源代码交给vm模块执行，并将结果缓存到Module._cache中，下次优先使用缓存
  5. 当遇到import()时调用esm加载器

es module的加载

  1. 代码交给v8运行，遇到import会交给esm加载器来进行处理（分静态和动态import）
  2. 查找模块的目标文件，对于文本代码直接读取文件内容
  3. 对于commonjs模块会用cjs的加载器来执行，缓存到CJSModule._cache
  4. 递归上述加载过程处理

和谐相处需约法三章

  • 在commonjs中只能通过import()使用es模块
  • 在es模块中不可以直接使用require，module，__filename等
  • 在es模块中import cjs模块时默认情况下只能引用默认导出
    ❌ import { a } from './a'
    ✅ import a from './a'
  • import不可省略文件后缀

package如何支持两种模块

  • 对于打包工具来讲，package.json中的 "module" 字段可以指定esm格式的入口文件
  • 对于node来讲，可以使用 "exports" 字段来指定别名来实现
    {
  "type": "module",
  "exports": {
    ".": {
      "require": "./main.cjs",
      "default": "./main.js"
   }
  }
}
