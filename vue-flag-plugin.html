Webpack的几个概念

  • Entry: 入口文件，Webpack 在这里开始构建
  • Module: 模块，在 Webpack 里一切皆模块，一个模块对应着一个文件
  • Dependency: 依赖，包含了一个模块与其他模块的依赖关系
  • Chunk: 代码块，一个 Chunk 由多个模块组合而成，用于代码合并与分割
  • Asset: 最后生成的资源文件(css, js, html, image..)
  • Loader: 用来转换不同格式的模块，最终转换为合法的JS源码
  • Plugin: 插件用来监听Webpack发出的事件，做一些增强的工作

Webpack构建过程

  1. 处理配置（合并、校验、转换），生成 compiler 对象，设置插件，监听文件变化
  2. 生成 compilation 对象，分析入口，开始构建
  3. 从入口文件出发调用loader处理模块
  4. 通过AST分析依赖，递归上面的步骤直到处理完所有的模块
  5. 根据依赖关系和配置项对模块进行组装、拆分、优化，生成chunk
  6. 对chunk通过代码模板生成最终的源代码
  7. 输出所有生成的资源文件，一次编译完成

loader能做什么

  • 同步或者异步处理当前模块[String source or Buffer(raw)]
  • 解析或者加载其他模块(resolve, loadModule)
  • 生成或更新Source Map
  • 管理依赖(addDependency)
  • 输出文件(emitFile)
  • 例子：raw-loader

pitch loader

有时候我们不关心模块的内容而只关心请求本身，此时可以用pitch-loader来实现

vue-loader内部会注入一个pitch loader来实现模版的编译和css scoped等特性

plugin能做什么

除了转换模块之外的几乎所有事情，例子：RemoveEmptyChunksPlugin

  • 处理Module, Chunk, Assets
  • 处理JS源码和生成代码
  • 处理构建信息

webpack = 插件？

  • webpack 本质上是基于 tapable 的一个事件系统
  • plugin 通过注册 webpack 暴露出来的 hooks 完成大部分的任务
  • 这些 hooks 的回调中包含 Module, dependency-tree, chunk, asset 以供 plugin 介入编译构建过程

compiler & compilation

  • compiler和compilation是webpack最重要的两个对象，它们都是tapable的实例。它们注册了webpack大部分的事件，整个编译构建过程都是围绕它们展开的
  • compiler: webpack主引擎，它负责解析配置，挂载插件以及开启compilation  compiler的hooks
  • compilation: 编译对象，表示一个完整的编译构建过程负责模块加载，封闭，优化，代码分块，哈希，重建等 当修改代码时，webpack会重启compilation的生命周期 包含着所有的module,chunk,assets等数据  compilation的hooks

插件用法

插件的执行过程

  1. 检查并处理选项，如果开启 watch，则监听flag文件的变化
  2. 将模板处理方法注入到 vue-loader 的选项中
  3. 如果需要忽略文件，则将ignore file loader注入到webpack中
  4. 应用 DefinePlugin 或者 ProvidePlugin
  5. 对样式的处理需要手动应用postcss插件

过滤代码需要考虑三种文件类型

  • 代码（js,ts,coffee...）
  • 模版 (只能用于SFC, html,pug,haml...)
  • 样式 (css,less,scss,stylus...)
关于脚本代码

生产模式使用 webpack.DefinePlugin

DefinePlugin会替换代码中flag变量为boolean

然后压缩工具会识别这些dead code并移除它们

关于脚本代码

watch模式下使用webpack.ProvidePlugin

它可以提供全局变量（namespace）访问flag

通过动态生成代码我们甚至可以介入到运行时来检查flag变量

关于样式

使用 postcss 插件来转换标准的@supports语法

/* css source code */
@support (--flag: A) {
  .this-a { color: #aaa; }
}
@support not (--flag: A) {
  .this-not-a { color: white; }
}
/* if A == true, after compiled */
.this-a { color: #aaa; }

为什么选择 @supports 的写法？

  • @supports 和自定义属性都是标准的css规范
  • 写法在现有的css预处理格式中都是兼容的
  • 从语义(supports, not, and, or)上来讲比较符合插件的功能

关于模版

通过指令来实现

<div v-if-flag="A">aaa</div>
<div v-elif-flag="B">bbb</div>
<div v-else>ccc</div>
<!-- after -->
<div>ccc</div>

模板的处理过程

  1. 如何介入对模板的处理过程

    vue的模板编译器提供了钩子(transformNode)可以介入模板AST的处理（通过compilerOptions.modules传入vue-loader）

    不能用directive选项，因为此时AST已经处理完毕，无法删除节点

  2. 在preTransformNode中收集v-*-flag的值，在postTransformNode中移除或保留对应的AST节点code

有时需要根据flag忽略一些模块

基于ignore file loader过滤，它是个pitch loader，因为此时我们不关心模块的内容，也不需要其他 loader 处理

如何实现开发时flag的热更新

  1. 监听flag文件变化，获取到新的flag值
  2. 将新的值序列化生成一个模块并写入一个缓存文件
  3. 将 namespace 和缓存文件的地址传入 webpack.ProvidePlugin
  4. 对于 css 和 template，将上述缓存添加到依赖树中，并且将在每次计算flag时重新读取值

注意事项

  • flag 指令不要和condition指令以及slot指令用在同一个节点上
  • 所有flag变量出现的地方都可以是表达式，但只能包含flag变量
  • @supports 可以嵌套，里面只能写css规则，不能直接写属性
